在日常开发中，我们经常会遇到这样的场景：

- 本地启动失败，终端输出几百行日志
- CI/CD 构建失败，需要从大量流水线日志中找错误
- 生产事故发生，多个服务同时报错
- Sentry、Nginx、Node.js、数据库日志混在一起
- 真正有价值的信息只有几行，却被大量 INFO、重复错误和调用栈淹没

这类任务的难点，往往不是“没有日志”，而是：

> 日志太多，真正重要的信息太少。

本篇文章将结合 Claude Code 的 Sub-Agent 机制和一个真实的日志分析示例，完成一个 `log-analyzer` 日志分析子代理。

我们不会只停留在配置文件讲解，而是会真正分析项目中的三类日志：

- 应用日志 `app.log`
- 错误日志 `error.log`
- 访问日志 `access.log`

最终理解：

- 为什么日志分析适合交给 Sub-Agent
- `Read`、`Grep`、`Glob`、`Bash` 分别负责什么
- 日志分析器是如何从原始日志中恢复事故时间线的
- 如何区分“事实”“关联”和“推测根因”
- 如何把 Demo 升级成企业级日志分析 Agent

## 一、为什么要单独创建日志分析子代理

先看一个常见问题。

假设你让 Claude Code 分析生产日志：

> 帮我分析 `logs/` 目录，看看系统为什么报错。

如果由主 Agent 直接完成，它可能会：

```text
读取 app.log
读取 error.log
读取 access.log
搜索所有 ERROR
搜索所有 WARN
统计重复错误
分析调用链
生成报告
```

功能上没有问题。

问题是，这些日志可能有几千行甚至几百万行。

当大量原始日志进入主对话后，主上下文会混入：

```text
业务需求
历史讨论
代码分析
INFO 日志
WARN 日志
ERROR 日志
调用栈
重复报错
访问记录
最终结论
```

后面你再让 Claude 修改代码、设计方案、提交 PR 时，主 Agent 仍然背着这些日志。

这就是典型的高噪声任务。

文章给出的核心思路是：

```text
主 Agent
    ↓ 委派日志分析任务
log-analyzer 子代理
    ↓
读取和分析大量日志
    ↓
只返回关键结论、证据和建议
```

原始噪声留在子代理上下文中，主对话只接收精炼结果。日志分析项目本身也明确把目标定义为：从冗长日志中完成筛选、分类和总结。

## 二、什么是日志场景中的“信噪比”

这里的“信噪比”可以理解为：

```text
真正影响决策的信息
────────────────
全部日志内容
```

例如原始日志有 10,000 行，你真正关心的是：

```text
1. 最早出现了什么问题？
2. 哪类错误最频繁？
3. 哪些接口受到了影响？
4. 哪个错误可能是根因？
5. 下一步先检查什么？
```

假设最后只需要 20 行结论，那么这个任务的信噪比就很低。

日志分析天然适合子代理，因为它通常具有几个特点：

- 输入量很大
- 重复内容很多
- 需要筛选关键词
- 需要恢复时间顺序
- 需要关联多个日志文件
- 主 Agent 最终只需要结论

但要注意：

> 日志多，不代表必须使用子代理；关键在于主 Agent 是否需要保留全部原始内容。

如果某次排查只涉及 10 行关键调用栈，直接在主对话分析即可。不要为了使用 Sub-Agent 而使用 Sub-Agent。

## 三、实战项目结构

目录结构如下：

```text
log-analyzer/
├── logs/
│   ├── app.log
│   ├── error.log
│   └── access.log
├── .claude/
│   └── agents/
│       └── log-analyzer.md
└── README.md
```

将三个日志文件划分为不同职责：

| 文件           | 内容                          |
| ------------ | --------------------------- |
| `app.log`    | 应用完整运行过程，包含 INFO、WARN、ERROR |
| `error.log`  | 错误详情、请求信息、堆栈和基础设施状态         |
| `access.log` | HTTP 请求、状态码、响应时间、用户和 IP     |

项目 `README` 也明确说明，这三个文件分别对应混合应用日志、错误日志和访问日志。

这三个文件不是简单重复，而是分别回答不同问题。

`app.log` 回答：

> 错误发生之前，系统正在做什么？

`error.log` 回答：

> 错误本身的详细证据是什么？

`access.log` 回答：

> 这个错误影响了哪些真实请求？

这也是企业排查问题时非常重要的思路：

> 不要只盯着一份错误日志，要把应用行为、错误证据和用户请求放到同一时间线上。

## 四、创建日志分析子代理

项目中的配置文件是：`.claude/agents/log-analyzer.md`

核心配置文件如下：

```markdown
---
name: 日志分析器
description: 分析日志文件并提取可操作的洞察。在排查问题或调查事故时使用。
tools: 读取文件, 文本检索(Grep), 文件查找(Glob), 命令行执行(Bash)
model: sonnet
---

你是一位资深站点可靠性工程师（SRE），专精于日志分析与事故调查。
```

完整配置定义了四个执行阶段：

```text
1. 找到日志文件
2. 扫描 ERROR、WARN 和异常
3. 分析重复模式与关联
4. 输出根因和可操作建议
```

这些流程在 Agent 配置中被明确写出。

### 1. name：子代理名称

```yaml
name: log-analyzer
```

它是子代理的唯一标识。

显式调用时可以写：

```text
让 log-analyzer 分析 logs/ 目录中的问题。
```

名称应该做到：

- 简短
- 能体现职责
- 不和其他 Agent 混淆

`log-analyzer` 比下面这些名称更合适：

```text
helper
debug-agent
analysis-tool
```

因为它直接告诉主 Agent：

```text
我是专门分析日志的。
```

### 2. description：能力和使用时机

```yaml
description: 分析日志文件并提取可操作的洞察。在排查问题或调查事故时使用。
```

这段 description 包含了两部分。

它能做什么：

```text
分析日志并提取可操作洞察
```

什么时候调用：

```text
故障排查或事故调查时使用
```

这非常重要，因为自然语言触发子代理时，主 Agent 会参考 description 进行路由。

例如用户说：

```text
分析一下 10 点到 11 点之间系统为什么大量报错。
```

虽然没有明确写 `log-analyzer`，主 Agent 仍有可能根据 `description` 选择它。

但需要保持清醒：

> 配置了 description，不等于自然语言一定百分之百触发子代理。

在开发和验证阶段，建议显式调用。

## 五、四个工具分别有什么作用

日志分析器拥有以下工具：

```yaml
tools: Read, Grep, Glob, Bash
```

把它定义为“只读工具加 `Bash`”，主要通过 `grep`、`awk` 等命令处理日志，同时不提供 `Edit` 和 `Write`。

### 1. Glob：先找到日志文件

`Glob` 用来按照文件路径和名称模式查找文件。
例如：

```text
logs/*.log
**/*.log
logs/error-*.log
```

假设项目里有：

```text
logs/app.log
logs/error.log
logs/access.log
logs/archive/error-2024-01-14.log
```

使用：

```text
**/*.log
```

就可以找到全部日志文件。

`Glob` 回答的是：

> 有哪些文件可能需要分析？

它主要找文件名和路径，而不是找文件里的内容。

### 2. Grep：从日志内容中筛选关键行

`Grep` 用于搜索文件内容。

例如：

```bash
grep "ERROR" logs/*.log
```

寻找所有错误。

```bash
grep "WARN" logs/*.log
```

寻找所有警告。

```bash
grep -i "timeout" logs/*.log
```

寻找超时相关内容。

`Grep` 回答的是：

> 哪些日志行包含我关心的内容？

例如用户说：

> 找出所有数据库超时。

日志分析器就不需要完整阅读每一行，而可以先搜索：

```bash
grep -i "database connection timeout" logs/*.log
```

### 3. Read：读取关键上下文

`Grep` 能定位关键行，但只看一行往往不够。

例如搜索到：

> 09:03:00 ERROR Database connection timeout

还需要读取它前后的日志：

```text
09:02:30 Slow query detected
09:02:31 Connection pool running low
09:03:00 Database connection timeout
09:03:00 API request failed
09:03:01 Reconnection successful
```

这时就需要 `Read`。

`Read` 回答的是：

> 这个错误发生前后，系统还发生了什么？

所以通常是：

```text
Glob 找文件
→ Grep 找异常
→ Read 看上下文
```

### 4. Bash：执行统计和文本处理命令

`Bash` 用来执行：

```bash
grep
awk
sort
uniq
wc
head
tail
```

项目配置中给出了一个快速扫描示例：

```bash
grep -c "ERROR" *.log
```

统计每个日志文件的错误数量。

以及：

```bash
grep "ERROR" *.log |
cut -d']' -f2 |
sort |
uniq -c |
sort -rn
```

用于尝试统计重复错误模式。

不过这里需要指出一个实际问题：

> 这条命令适合作为教学示意，但在真实复杂日志中不一定可靠。

因为真实日志可能存在：

- 多行堆栈
- JSON 日志
- 不同时间格式
- 不同字段顺序
- 错误中带动态 ID
- 相同错误包含不同参数
- 一条异常跨越几十行

例如：

```text
Database timeout for user 123
Database timeout for user 456
```

字符串不同，但本质上可能是同一类错误。

真实项目里通常需要：

- 正则归一化
- 删除时间戳和动态 ID
- 按错误类型聚合
- 按请求 ID 串联
- 使用结构化 JSON 日志

## 六、为什么日志分析器没有 Edit 和 Write

日志分析器的职责是：

```text
找日志
筛日志
分析日志
输出报告
```

不是：

```text
发现问题后直接修改代码
```

因此它没有：

```yaml
Edit
Write
```

这是正确的权限设计。

如果日志分析器拥有修改权限，它可能执行：

```text
发现 TypeError
→ 猜测空值问题
→ 直接修改产品服务
→ 继续分析其他日志
```

这会产生职责混乱：

- 分析和修复混在一起
- 推测可能直接变成代码修改
- 难以审查修改依据
- 错误归因可能导致错误修复

合理流程应该是：

```text
主 Agent
    ↓
log-analyzer：负责分析事故
    ↓
主 Agent 审核证据
    ↓
bug-fixer：负责修改代码
    ↓
test-runner：负责执行验证
    ↓
code-reviewer：负责代码审查
```

这和真实团队很像：

```text
SRE 定位问题
开发修复问题
测试验证问题
Reviewer 审查改动
```

职责越清晰，自动化越可控。

## 七、为什么日志分析器使用更强的模型

项目中使用：

```yaml
model: sonnet
```

原因不是“日志很多”，而是日志分析需要推理。

测试运行器通常处理：

```text
PASS
FAIL
测试数量
错误堆栈
断言差异
```

日志分析器则需要处理：

```text
错误模式识别
时间线恢复
多组件关联
根因和症状区分
事故影响评估
处理优先级排序
```

例如日志中同时出现：

```text
数据库连接池耗尽
API 503
Redis 连接断开
库存不足
接口 TypeError
SMTP 连接失败
```

它必须判断：

- 哪些错误属于同一个问题
- 哪些错误互相独立
- 哪个错误最早发生
- 哪个错误影响最大
- 哪个只是业务异常
- 哪个是真正的基础设施故障

这不是单纯的关键词匹配。

但也不能机械地理解为：

```text
测试运行器固定用小模型
日志分析器固定用大模型
```

真正应该根据任务复杂度选择模型：

```text
执行命令、统计结果、套模板
→ 较轻模型

模式识别、跨文件关联、时序推理
→ 较强模型

复杂事故、跨服务架构推理
→ 更强模型或多个专业 Agent
```

## 九、实战：让 log-analyzer 分析项目日志

显式调用日志分析子代理：

```text
让 log-analyzer 分析 logs/ 目录下的日志，
找出主要问题、时间线和可能原因，不要修改任何文件。
```

子代理会按照配置中的流程完成分析：

```text
Glob 找到 app.log、error.log、access.log
        ↓
Grep 筛选 ERROR、WARN、timeout 等关键词
        ↓
Read 查看异常前后的上下文
        ↓
Bash 统计错误数量和重复模式
        ↓
整理事故时间线并输出结构化报告
```

项目中的日志大致包含以下问题：

- 数据库连接池耗尽，多次导致接口返回 503
- 订单事务发生数据库死锁
- 产品接口访问了 undefined.price
- SMTP 服务连接失败，导致日报任务失败
- Redis 连接中断并重连失败
- 异常登录和请求限流事件

但子代理不应该把所有原始日志重新输出，而应压缩成类似下面的结果：

```markdown
## Log Analysis Report

### Executive Summary

数据库连接池分别在 09:03 和 10:25 两次耗尽，
导致多个 API 请求超时并返回 503。

此外还存在 Redis 连接中断、订单事务死锁、
产品接口空值异常和 SMTP 服务不可用等问题。

### Critical Issues

1. 数据库连接池耗尽
   - 首次发生：09:03
   - 再次发生：10:25
   - 影响：多个接口返回 503
   - 建议：检查慢查询、连接释放和连接池监控

2. Redis 连接失败
   - 发生时间：10:45
   - 现象：连接中断且重连失败
   - 建议：检查 Redis 服务和接口降级策略

### Application Errors

- 订单事务死锁，导致订单创建失败
- 产品接口缺少空值处理
- SMTP 不可用，导致日报任务失败

### Security Events

- admin 账户出现连续失败登录
- 搜索接口触发请求限流
```

这个实战最重要的，不是记住日志里发生了哪些错误，而是观察子代理如何完成信息压缩：

```markdown
原始日志
→ 错误筛选
→ 分类聚合
→ 时间线整理
→ 输出结论和建议
```

然后保留一个简短的“严谨性提醒”即可：

## 十、最终应该输出什么样的报告

```markdown
## Log Analysis Report

### Executive Summary

系统存在两个优先级较高的问题：

1. 数据库连接池在 09:03 和 10:25 两次耗尽，造成多个接口超时并返回 503。
2. 10:45 Redis 连接丢失且重连失败，同一时刻 Dashboard 接口返回 503。

此外还存在订单事务死锁、产品接口空值异常、SMTP 服务不可用，以及两类安全/限流事件。

### Critical Issues

#### 1. 数据库连接池耗尽

- 首次发生：09:03:00
- 再次发生：10:25:00
- 证据：Pool status 均为 0/10 available
- 影响：
  - GET /api/products → 503
  - GET /api/products?category=electronics → 503
  - GET /api/orders/count → 503
- 前置信号：
  - 慢查询
  - 连接池仅剩 2/10 可用连接
- 可能原因：
  - 慢查询长期占用连接
  - 连接未释放
  - 突发并发流量
  - 连接池容量不足
- 建议：
  1. 检查慢查询和连接占用时间
  2. 排查连接泄漏
  3. 增加连接池使用率监控
  4. 不要在确认根因前只盲目扩大连接池

#### 2. Redis 连接中断

- 首次发生：10:45:30
- 现象：ECONNRESET，随后 ECONNREFUSED
- 关联影响：Dashboard 同时返回 503
- 结论：存在较强时间关联，但仍需验证 Dashboard 是否依赖 Redis
- 建议：
  1. 检查 Redis 实例和网络状态
  2. 检查 Dashboard 缓存依赖
  3. 实现 Redis 不可用时的降级策略

### Application Errors

#### 订单事务死锁

- 时间：09:07:46
- 涉及：orders、inventory
- 影响：POST /api/orders 返回 500
- 建议：统一锁顺序、缩短事务、增加死锁重试

#### 产品接口空值异常

- 时间：09:35:15
- 位置：ProductService.calculateTotal:45
- 错误：读取 undefined.price
- 影响：GET /api/products/999 返回 500
- 建议：追踪 req_abc123，验证查询结果并增加空值保护

#### SMTP 服务不可用

- 时间：10:00:15
- 错误：ECONNREFUSED
- 影响：每日报表任务在重试三次后失败
- 建议：检查 SMTP 服务、网络和凭据，并引入异步重试或补偿任务

### Security and Traffic Events

- 203.0.113.50 对 admin 连续尝试登录，账户锁定和限流已生效
- 192.168.1.100 对搜索接口发起 156～158 次/分钟请求，触发 429

### Timeline

- 09:02:30：出现慢查询
- 09:02:31：数据库连接池降至 2/10
- 09:03:00：连接池耗尽，API 返回 503
- 09:06:31：admin 账户因重复登录失败被锁定
- 09:07:46：订单事务死锁，创建订单失败
- 09:35:15：产品接口出现空值异常
- 10:00:15：SMTP 连接失败
- 10:00:16：报表任务失败
- 10:10:30：搜索接口触发限流
- 10:25:00：数据库连接池再次耗尽
- 10:45:30：Redis 连接丢失
- 10:45:35：Redis 重连失败

### Root Cause Assessment

已确认：
- 数据库连接池两次耗尽
- 订单事务发生死锁
- 产品接口存在未处理的空值访问
- SMTP 和 Redis 均发生连接失败

尚未确认：
- 数据库连接池耗尽的具体根因
- Dashboard 503 是否完全由 Redis 故障导致
- 产品对象为空的上游来源

### Recommended Priority

1. 先处理重复发生的数据库连接池耗尽
2. 排查 Redis 故障和 Dashboard 降级能力
3. 修复产品接口空值问题
4. 优化订单事务死锁
5. 恢复邮件服务并补发失败报表
6. 审查异常登录和搜索流量来源
```

这类输出才真正实现了：

```text
大量原始日志
    ↓
结构化事实
    ↓
问题分类
    ↓
时间线
    ↓
证据支持的根因判断
    ↓
有优先级的处理建议
```

## 十一、输出格式为什么这么重要

子代理的输出会成为主 Agent 的输入。

如果日志分析器返回的是：

```text
数据库有问题。
Redis 也有问题。
代码可能有 Bug。
建议检查一下。
```

它虽然很短，但没有任何决策价值。

一个好的输出需要遵循三层结构。

### 第一层：结论先行

```text
数据库连接池两次耗尽，是当前最高优先级问题。
```

主 Agent 一眼知道先处理什么。

### 第二层：关键证据

```text
09:03 和 10:25 均出现 Pool status 0/10，
并对应多个 503 请求。
```

主 Agent 知道结论从哪里来。

### 第三层：补充分析

```text
可能原因包括慢查询、连接泄漏和连接池配置不足，
需要进一步检查慢查询和连接占用。
```

主 Agent 知道下一步如何验证。

这就是：

```text
结论
→ 证据
→ 行动
```

而不是：

```text
堆砌日志
→ 大段解释
→ 最后才说结果
```

## 十二、最容易犯的错误：把时间关联当成因果关系

日志分析器最大的风险不是看不到错误，而是过度推理。

例如：

```text
09:03 数据库超时
09:07 订单死锁
09:35 产品 TypeError
```

不能直接写：

```text
数据库超时引发了后面所有问题。
```

因为：

- 数据库超时和死锁可能属于不同问题
- TypeError 更像代码空值处理缺失
- 时间靠近不等于因果关系
- 缺乏请求 ID、Trace ID 或调用链证据

因此企业级日志报告应该区分三种表述。

### 已确认事实

```text
10:25 数据库连接池为 0/10，两个 API 返回 503。
```

### 可能关联

```text
Redis 连接失败和 Dashboard 503 同时发生，存在较强关联。
```

### 待验证假设

```text
数据库连接池耗尽可能与慢查询或连接泄漏有关，
需要结合数据库监控和连接池指标确认。
```

这是一个日志 Agent 是否可靠的核心分界线。

## 十三、原项目配置还可以怎样改进

原配置适合作为入门教学，但如果用于企业项目，还不够。

可以在输出中增加：

```text
Evidence
Confidence
Known Facts
Hypotheses
Missing Data
Affected Services
Affected Requests
Suggested Verification
```

例如改进后的输出模板：

```markdown
## Incident Analysis

### Summary
[一句话描述最高优先级问题]

### Confirmed Facts
- [事实 + 时间 + 日志文件 + 关键证据]

### Correlated Events
- [两个事件之间的关联及依据]

### Hypotheses
1. [可能原因]
   - Confidence: High / Medium / Low
   - Supporting evidence:
   - Contradicting evidence:
   - Verification:

### Impact
- Affected endpoints:
- Error responses:
- Affected users:
- Duration:

### Timeline
- [timestamp] [event]

### Recommended Actions
1. Immediate:
2. Short term:
3. Long term:

### Missing Evidence
- [还需要哪些日志、指标或 Trace]
```

这个模板比单纯的“Root Cause Analysis”更严谨。

因为日志通常只能帮助提出高质量假设，不一定能单独证明最终根因。

## 十四、企业项目还必须补齐哪些能力

这个仓库是一个教学 Demo，日志规模只有几百行，并且日志格式相对整齐。

真实项目中还需要处理下面的问题。

### 1. 超大日志不能全部读取

生产日志可能几 GB，不能让 Agent 直接 `Read` 全部文件。

应该先用命令筛选：

```bash
grep "ERROR" app.log | tail -n 200
```

或者按时间过滤：

```bash
awk '$0 >= "2026-07-12 10:00:00" && $0 <= "2026-07-12 11:00:00"' app.log
```

更合理的流程是：

```text
先统计
→ 再过滤
→ 再聚合
→ 最后只读取关键上下文
```

### 2. 必须设置执行范围

不要允许日志 Agent 无限制扫描：

```text
/
```

或者：

```text
/home
```

应限制在：

```text
项目 `logs/`
指定容器日志
指定时间区间
指定服务
指定最大行数
```

否则可能：

- 扫描大量无关文件
- 读取敏感信息
- 执行时间过长
- 消耗大量 token

### 3. 日志中可能包含敏感信息

例如示例日志里已经出现：

```text
数据库连接地址
用户名
用户 ID
IP 地址
邮箱地址
```

真实系统中还可能包含：

```text
Token
Cookie
手机号
身份证号
请求参数
订单数据
用户隐私
数据库密码
```

因此日志 Agent 需要脱敏规则：

```text
Authorization: Bearer ***
Cookie: ***
password=***
phone=138****1234
```

否则分析报告本身可能成为数据泄漏源。

### 4. 必须处理多行异常

Java、Python、Node.js 的调用栈经常跨越多行。
不能只执行：

```bash
grep ERROR
```

因为它可能只拿到第一行，丢失真正的堆栈。
需要读取：

```text
错误行前后 N 行
从异常开始到下一条时间戳之间的全部内容
```

或者优先使用结构化日志。

### 5. 最好结合 Trace ID

单纯靠时间关联，可靠性有限。
如果日志中有：

```text
trace_id
request_id
span_id
user_id
```

就可以准确串联：

```text
Nginx 请求
→ API 网关
→ Node.js 服务
→ 数据库查询
→ Redis 请求
→ 下游服务
```

例如示例里已经有：

```text
Request ID: req_abc123
```

但其他日志没有继续携带这个 ID，所以无法完成真正的全链路关联。

企业级日志系统应该尽量让同一请求在所有组件中携带统一 Trace ID。

## 十五、结合前端项目，可以做哪些日志子代理

你是一名前端开发工程师，日志分析器并不只适用于后端生产日志。

它可以迁移到很多真实场景。

### 1. Vite 构建日志分析器

输入：

```text
pnpm build 的 500 行输出
```

输出：

```text
构建是否成功
第一个阻断错误
文件和行号
TypeScript 错误
依赖错误
普通 warning
建议处理顺序
```

### 2. ESLint 日志分析器

输入：

```text
几百条 ESLint 报错
```

输出：

```text
错误总数
按规则分类
按目录分类
可自动修复数量
阻断构建问题
建议优先处理规则
```

### 3. Electron 日志分析器

可以分别分析：

```text
main process 日志
renderer 日志

```

然后输出：

```text
错误发生在哪个进程

是否发生 renderer crash
是否是 preload 暴露失败
是否是主进程异常
```

## 十六、Sub-Agent 的正确协作链路

最终，一个成熟的 AI 开发流程可以设计成：

```text
用户提出问题
    ↓
主 Agent 拆解任务
    ↓
log-analyzer 分析日志
    ↓
输出事实、时间线、可能根因
    ↓
impact-analyzer 分析影响范围
    ↓
bug-fixer 修改代码
    ↓
test-runner 执行测试
    ↓
code-reviewer 审查改动
    ↓
主 Agent 汇总结论
```

每个子代理只承担一个明确职责：

| Agent           | 主要职责      | 是否修改代码 |
| --------------- | --------- | -----: |
| log-analyzer    | 从日志中定位问题  |      否 |
| impact-analyzer | 分析影响链路    |      否 |
| bug-fixer       | 修改代码      |      是 |
| test-runner     | 运行并总结测试   |      否 |
| code-reviewer   | 审查安全性和正确性 |      否 |

这才是“分而治之”的真正价值。
不是创建很多 Agent 看起来很高级，而是:

> 把不同风险、不同输入、不同权限的任务分开。

## 十七、总结

通过这个日志分析器项目，我们建立了下面几层认识。

第一，日志分析不是简单搜索 ERROR

真正有价值的分析需要：

```text
定位首个异常
恢复事故时间线
聚合重复错误
关联不同日志
区分根因和症状
评估影响范围
给出验证步骤
```

第二，子代理的核心价值是噪声隔离。

```text
大量日志留给 log-analyzer
主 Agent 只接收结论和证据
```

第三，工具权限需要与职责匹配。

```text
Glob：找日志文件
Grep：筛关键内容
Read：查看错误上下文
Bash：统计、过滤和聚合
```

日志分析器不需要 Edit 和 Write，因为分析者不应该擅自修改代码。

第四，模型选择取决于推理难度。

日志分析涉及模式识别、时间关联和根因推理，因此通常比单纯的测试结果汇总需要更强的模型。

第五，最重要的不是“输出一个看起来合理的根因”，而是：

```text
事实是什么
证据在哪里
哪些只是关联
哪些仍是猜测
下一步如何验证
```

一个可靠的日志分析 Agent，不应该只是会说：

```text
我认为数据库有问题。
```

而应该能够说：

```text
09:02:31 连接池只剩 2/10，
09:03:00 变成 0/10，
同一时刻接口等待 5 秒后返回 503。
可以确认连接池耗尽，但耗尽原因仍需结合慢查询、
连接泄漏和数据库监控继续验证。
```

这才是能真正用于工程实践的分析结果。
