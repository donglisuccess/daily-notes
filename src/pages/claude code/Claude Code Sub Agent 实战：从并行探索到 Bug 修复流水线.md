当我们第一次使用 Claude Code 时，通常会把一个完整任务直接扔给主对话：

> 帮我阅读项目、定位问题、分析原因、修改代码、运行测试，最后告诉我结果。

对于一个小项目、一个简单 Bug，这种方式没有问题。

但当项目逐渐变大，任务逐渐复杂，你会遇到三个明显的痛点：

1. 慢：很多互不依赖的任务仍然被一个接一个执行。
2. 乱：搜索结果、代码片段、日志和测试输出全部挤在主对话里。
3. 不可控：Claude 从定位问题直接跳到修改代码，你很难判断它到底在哪一步走偏了。

Sub Agent 真正的价值，并不是“多创建几个 AI 角色”，而是把原本混乱的工作过程，改造成一套可以拆分、编排、审查和回退的工程流程。

这篇文章通过两个实战项目，重点讲清楚两种模式：

- 并行探索：解决多个独立任务同时执行的问题。
- 流水线编排：解决多个依赖阶段按顺序协作的问题。

核心判断只有一句话：

> 任务独立就并行，结果依赖就流水线。

## 一、先看痛点：为什么一个主对话不够用

假设你刚接手一个陌生的 Node.js 后端项目，里面包含：

```text
src/
├── auth/
├── database/
├── api/
├── payment/
├── order/
└── notification/
```

你需要理解：

- 用户如何登录；
- JWT 如何生成和校验；
- 数据库有哪些核心表；
- API 如何鉴权；
- 请求经过哪些中间件；
- 各模块之间如何传递用户身份。

最直觉的做法是直接对 Claude Code 说：

> 帮我完整分析这个项目的架构。

Claude Code 接下来可能会：

1. 搜索大量文件；
2. 阅读认证模块；
3. 阅读数据库模块；
4. 阅读 API 模块；
5. 输出大量中间搜索结果；
6. 尝试在有限上下文里总结。

问题不是 Claude 看不懂，而是整个过程缺乏结构。

### 1. 串行执行浪费时间
认证、数据库、API 三个模块，在初始探索阶段往往没有强依赖。

但如果让一个主对话处理，它通常还是按照顺序执行：

```text
auth → database → api → 总结
```

假设每个模块需要 30 秒，理论总耗时接近 90 秒。

实际上，这三个任务完全可以同时开始。

### 2. 主对话被低价值信息污染
阅读一个模块时，Claude 可能执行很多操作：

```text
Glob 搜索文件
Grep 搜索关键字
Read 阅读入口文件
Read 阅读实现文件
继续 Grep 调用方
```

这些过程是必要的，但不一定值得全部保留在主对话中。

主对话真正需要的应该是：

```text
认证模块使用 JWT。
Access Token 有效期 30 分钟。
Refresh Token 保存在数据库。
API 通过 authMiddleware 校验用户身份。
管理员权限依赖 token 中的 role 字段。
```

换句话说：

> 主对话需要的是“报告”，而不是“调查过程”。

### 3. 读权限和写权限混在一起
如果同一个 Agent 同时负责：

- 搜索代码；
- 判断根因；
- 修改文件；
- 执行命令；

那么它从一开始就拥有较高权限。

这在复杂项目中并不稳妥。

更合理的过程应该是：

```text
先只读定位
    ↓
再只读分析
    ↓
人工确认方向
    ↓
授予修改权限
    ↓
最后运行测试
```

这正是 Sub Agent 编排开始有工程价值的地方。

## 二、Sub Agent 不是“另一个聊天窗口”

很多人理解 Sub Agent 时，容易停留在表面：

> Sub Agent 就是给 Claude 定义几个不同角色。

Sub Agent 就是给 Claude 定义几个不同角色。

这不够准确。

一个工程化的 Sub Agent，至少包含四部分。

```yaml
---
name: auth-explorer
description: 探索认证相关代码
tools: Read, Grep, Glob
model: haiku
---
```

下面再配合具体的任务说明、职责边界和输出格式。

它实际上定义的是：

| 配置            | 解决的问题        |
| ------------- | ------------ |
| `name`        | 这个执行单元叫什么    |
| `description` | 主对话什么时候应该调用它 |
| `tools`       | 它允许做什么       |
| `model`       | 任务需要什么能力和成本  |
| Prompt 正文     | 它具体怎么工作      |
| Output Format | 它必须交付什么结果    |

所以，Sub Agent 不只是“角色设定”，而是一个**受约束的工作单元**。

可以把它理解成团队里的专业岗位：

```text
主对话：项目负责人
Sub Agent：专业执行人员
Tools：岗位权限
Prompt：岗位说明书
Output Format：交付标准
```

真正的重点不是创建多少个 Agent，而是：

> 是否把正确的任务，交给了权限合适、边界清楚、输出稳定的执行单元。

## 三、实战一：用并行探索快速理解陌生项目

项目将代码划分为三个主要领域：

```text
parallel-explore/
├── src/
│   ├── auth/
│   ├── database/
│   └── api/
└── .claude/agents/
    ├── auth-explorer.md
    ├── db-explorer.md
    └── api-explorer.md
```

项目本身的设计目标很明确：让三个只读子代理分别探索认证、数据库和 API，最后由主对话进行综合。

### 1. 为什么这个任务适合并行
先判断三个任务之间是否存在前置依赖。

**auth-explorer**

负责回答：

```text
用户如何登录？
Token 如何生成？
Token 在哪里校验？
Session 如何管理？
权限如何判断？
```

**db-explorer**

负责回答：

```text
有哪些核心数据模型？
表之间是什么关系？
数据库如何初始化？
迁移脚本如何执行？
```

**api-explorer**

负责回答：

```text
有哪些路由？
请求经过哪些中间件？
Controller 如何调用业务代码？
错误如何统一处理？
```

每个 Agent 都可以先独立完成自己的模块级调查。

它们不需要等待另一个 Agent 的结果，因此适合并行。

注意这里有一个严格前提：

> “能够分别调查”不等于“最终没有关联”。

认证、数据库和 API 最终肯定存在跨模块关系，但这些关系可以在三份报告返回后，由主对话统一综合。

所以完整结构不是单纯的并行，而是：

```text
                    ┌─ auth-explorer ─┐
用户任务 → 拆分任务 ├─ db-explorer ───┤ → 主对话综合 → 整体架构
                    └─ api-explorer ──┘
```

这叫作：

> Fan-out → Fan-in，扇出后聚合。

### 2. 编写 auth-explorer

在项目中创建：

> .claude/agents/auth-explorer.md

配置可以写成：

```markdown
---
name: auth-explorer
description: 探索和分析认证相关代码。适用于登录、JWT、Session、权限和用户身份传递问题。
tools: Read, Grep, Glob
model: haiku
---

你是一位认证模块分析专家。

## 职责范围

只分析以下内容：

- 登录和退出流程
- JWT 的生成与校验
- Access Token 与 Refresh Token
- Session 管理
- 用户身份传递
- 角色和权限校验
- 密码处理

不要深入分析数据库性能、普通业务接口或与认证无关的模块。

## 执行步骤

1. 使用 Glob 搜索认证相关文件：
   - `**/auth/**`
   - `**/*auth*`
   - `**/*login*`
   - `**/*session*`
   - `**/*jwt*`

2. 使用 Grep 搜索关键方法：
   - `login`
   - `verify`
   - `sign`
   - `token`
   - `permission`
   - `role`

3. 阅读入口文件和关键实现文件。

4. 梳理认证流程和身份传递路径。

## 输出格式

# 认证模块分析报告

## 模块概览

用 2～3 句话说明认证模块采用的整体方案。

## 登录流程

1. 客户端调用：
2. 服务端校验：
3. Token 生成：
4. Token 返回：
5. 后续请求校验：

## 关键文件

| 文件 | 关键函数 | 作用 |
|---|---|---|
| | | |

## Token 策略

- Token 类型：
- 生成位置：
- 校验位置：
- 有效期：
- 存储方式：
- 刷新机制：

## 权限流程

说明 role、permission 或用户身份如何传递和校验。

## 风险点

- 安全风险：
- 边界问题：
- 暂时无法确认的内容：

## 供主对话综合的信息

- 与数据库模块的连接点：
- 与 API 模块的连接点：
- 建议进一步核实的问题：
```

### 3. 为什么只给 Read、Grep、Glob

参考项目中的三个探索 Agent 都使用只读工具，并选择速度更快的模型。

这里涉及一个非常重要的权限设计原则：

> 能用只读权限完成的任务，不要提前给写权限。

三个工具分别负责不同层级的探索。

**Glob：找文件**

例如：

```text
**/auth/**
**/*jwt*
src/**/*.js
```

它适合回答：

> 哪些文件可能相关？

**Grep：找内容**

例如：

```text
verifyToken
jwt.sign
role
permission
```

它适合回答：

> 哪些代码中出现了这个函数或关键字？

**Read：读实现**

在定位到文件后，再阅读具体内容。

它适合回答：

> 这个函数究竟是怎么实现的？

三者结合起来就是：

```text
Glob 找文件
   ↓
Grep 找位置
   ↓
Read 读逻辑
```
探索任务不需要 Edit、Write 或 Bash。

如果一开始就给探索 Agent 写权限，不会让它“更聪明”，只会扩大误操作范围。

### 4. db-explorer 和 api-explorer 怎么写

不需要复制完全不同的 Prompt。

保留相同骨架，只替换三个部分：

1. 领域边界；
2. 搜索关键词；
3. 输出结构。

例如，db-explorer 应重点输出：

```markdown
## 核心数据模型

| 模型 | 文件 | 关键字段 | 关系 |
|---|---|---|---|

## 数据访问流程

业务代码如何调用数据库？

## 数据迁移

迁移文件在哪里？
如何执行？
是否存在初始化数据？

## 跨模块连接点

- 认证模块使用了哪些表？
- API 模块通过什么方式访问数据？
```

api-explorer 应重点输出：

```markdown
## API 入口

服务从哪个文件启动？

## 请求链路

请求
→ 路由
→ 中间件
→ Controller
→ Service
→ 数据库
→ 响应

## 路由清单

| Method | Path | Handler | Middleware |
|---|---|---|---|

## 跨模块连接点

- 哪些接口需要认证？
- 哪些接口访问数据库？
- 用户身份如何进入业务函数？
```

三个报告的结构不需要完全相同，但抽象粒度应该一致。

不能出现：

```text
auth-explorer 输出模块级架构
db-explorer 输出每一行 SQL
api-explorer 只输出三个文件名
```

否则主对话很难综合。

### 5. 启动并行探索

在 Claude Code 中，可以输入：

```text
请并行调用以下三个子代理：

1. auth-explorer：分析认证流程和权限传递
2. db-explorer：分析数据模型、数据访问和迁移机制
3. api-explorer：分析 API 路由、中间件和请求链路

要求：
- 三个任务相互独立执行
- 不修改任何代码
- 等三份报告全部返回后，再由主对话综合
- 最终输出：
  1. 项目整体架构
  2. 一条完整请求链路
  3. 认证、API、数据库之间的依赖关系
  4. 关键风险点
  5. 建议优先阅读的文件
```
参考项目推荐的调用方式也是同时启动三个 Explorer，再由主对话收集和综合结果。

这里有一个容易误解的地方：

> 创建了三个 Sub Agent，不代表它们一定会自动并行。

是否并行，取决于主对话如何拆分和调度任务。

因此 Prompt 中最好明确写出：

```text
- 并行调用
- 等待全部完成
- 最后统一综合
```

### 6. 主对话如何做综合

三个 Agent 返回的报告仍然只是三块局部信息。

主对话必须完成“连点成线”。

假设得到三条发现：

```text
auth-explorer：
JWT 中包含 userId 和 role。

db-explorer：
users 表存储 role，orders 表冗余存储 user_role。

api-explorer：
/admin/* 路由通过 requireRole('admin') 校验权限。
```

主对话不能只把三份报告拼接起来，而应该进一步分析：

```text
users.role
   ↓ 登录时读取
JWT.role
   ↓ 请求携带
authMiddleware 解码
   ↓ 写入 req.user
requireRole('admin')
   ↓ 控制 /admin/* 路由
```

同时还应提出风险：

```text
orders.user_role 是历史快照，还是实时权限字段？

如果 users.role 已修改，而旧订单中的 user_role 未同步，
两个字段可能产生语义冲突。
```

这才叫综合。

简单拼接不叫综合。

### 7. 并行探索最容易踩的坑

#### 坑一：任务其实存在前置依赖

例如：

```text
Agent A：先确定项目使用哪种认证方案
Agent B：根据认证方案检查对应安全漏洞
```

B 必须先拿到 A 的结果，因此不能并行。

正确方式是：

```text
A 确定认证方案
    ↓
B 检查安全问题
```

#### 坑二：职责边界重叠

例如 auth-explorer 和 api-explorer 都在完整分析鉴权中间件。

结果会造成：

- 重复读取；
- 重复消耗；
- 输出互相矛盾；
- 主对话难以判断谁更可信。

更合理的边界是：

```text
auth-explorer：
分析 token 如何生成、解析、形成用户身份。

api-explorer：
分析哪些路由使用鉴权中间件，以及调用顺序。
```

#### 坑三：把并行当成越多越好

将项目拆成 20 个 Agent，不一定比 3 个 Agent 更快。

你还需要承担：

- 调度成本；
- 结果汇总成本；
- 重复搜索成本；
- 上下文传递成本；
- 冲突消解成本。

合理的拆分标准不是文件数量，而是：

> 是否存在清晰、独立、可以单独交付结果的责任域。

#### 坑四：没有综合阶段

并行探索的最终产物不是三份报告，而是：

```text
三份局部报告
    ↓
跨模块关系
    ↓
整体数据流
    ↓
风险与疑问
```
没有综合阶段，并行探索只能得到碎片化知识。

## 四、实战二：把 Bug 修复拆成一条流水线

现在来看第二个项目：

```text
SubAgents/projects/05-bugfix-pipeline
```

它定义了四个 Sub Agent：

```text
bug-locator
    ↓
bug-analyzer
    ↓
bug-fixer
    ↓
bug-verifier
```
参考项目明确将 Bug 修复拆成定位、分析、修复、验证四个阶段，并由主对话负责串联。

为什么这里不能并行？

因为它们存在明确的结果依赖：

```text
不知道问题在哪里
    ↓
就无法准确分析根因

不知道根因是什么
    ↓
就无法设计可靠修复

还没完成修改
    ↓
就无法验证结果
```

所以这不是三个、四个独立任务，而是一条流水线。

## 五、第一阶段：Locator 只回答“问题在哪”

bug-locator 的任务不是修复，也不是给出最终根因。

它只负责缩小调查范围。

```yaml
---
name: bug-locator
description: 定位代码库中 Bug 最可能出现的位置，作为 Bug 修复流水线的第一阶段。
tools: Read, Grep, Glob
model: sonnet
---
```

Prompt 的核心约束：

```markdown
你只负责定位，不得修改代码，也不要直接给出修复方案。

## 输入

你可能收到：

- Bug 描述
- 错误信息
- 堆栈
- 复现步骤
- Sentry 事件
- 用户反馈

## 工作流程
1. 提取错误关键词、函数名和业务关键词。
2. 搜索相关文件和调用位置。
3. 阅读最可能的实现。
4. 追踪直接调用方和关键依赖。
5. 给出候选位置及证据。

## 输出格式

# Bug 定位报告

## 症状摘要

## 候选位置

| 优先级 | 文件 | 函数 | 行号 | 证据 | 置信度 |
|---|---|---|---|---|---|

## 最可能的位置

- 文件：
- 函数：
- 行号：
- 判断依据：

## 相关调用链

调用入口
→ 中间函数
→ 出错函数

## 已排除的位置

| 位置 | 排除原因 |
|---|---|

## Handoff：移交给 Analyzer

- Primary suspect：
- Symptoms to reproduce：
- Relevant call chain：
- Hypothesis：
- Already excluded：
- Related files：
- Remaining uncertainty：
```

这里最重要的一句话是：

> Locator 不要提供修复方案。

为什么？

因为“找到错误发生的位置”和“确认错误产生的根因”不是一回事。

例如某个前端错误：

```js
const suffix = name.split('.')[1]
```

Sentry 报错位置可能就在这里：

```text
Cannot read properties of undefined
```

但真正的根因可能是：

- 接口字段在某些数据中为空；
- 上游转换逻辑丢失了 name；
- 某种文件类型没有名称；
- 异步数据尚未完成加载；
- 调用方传错了对象。

Locator 应该说：

```text
异常发生在 name.split()。
name 来自接口响应中的 fileInfo.name。
有三个调用入口，其中预览接口可能返回空值。
```

而不是立刻说：

```js
(name || '').split('.')
```

后者已经进入修复阶段了。

## 六、第二阶段：Analyzer 只回答“为什么”

bug-analyzer 接收 Locator 的报告，继续分析根因。

```yaml
---
name: bug-analyzer
description: 根据定位报告分析 Bug 根本原因、影响范围和修复方向。
tools: Read, Grep, Glob
model: sonnet
---
```

它的核心任务不是重新做一遍全库搜索，而是围绕已经定位的范围深挖：

```markdown
## 输入要求

开始前确认已经收到：

- 具体文件
- 具体函数
- 行号范围
- Bug 症状
- 候选调用链
- 已排除位置

如果缺少关键输入，明确指出，不要假装已经确认根因。

## 分析重点

- 数据从哪里进入？
```

### 为什么 Analyzer 到 Fixer 之间必须停一下
这是整条流水线中最重要的人工审批点。

在此之前：

```text
Locator：只读
Analyzer：只读
```

接下来：

```text
Fixer：开始修改代码
```

这意味着系统将从“调查”进入“执行”。

一旦根因分析错了，Fixer 可能会做出一份逻辑正确、方向错误的修改。

例如真实根因是：

```text
接口偶尔返回 null。
```

但 Analyzer 错误判断为：

```text
前端初始化顺序有问题。
```

Fixer 可能会：

- 增加大量 loading 状态；
- 重构组件生命周期；
- 修改多个调用点；
- 引入无关复杂度；

而真正需要的可能只是修正接口契约或在数据边界统一处理。

因此建议使用下面的编排方式：

```text
Locator 自动完成
    ↓
Analyzer 自动完成
    ↓
暂停，展示根因报告
    ↓
人工确认
    ↓
Fixer 执行
    ↓
Verifier 自动验证
```

这比“全自动跑到底”稳得多。

这里我们比基础版本多加了两个工程控制：

- Analyzer 后人工审批；
- Verifier 失败后禁止 Fixer 盲目重试。

## 十、流水线真正的核心：交接契约

很多人配置完四个 Agent 后，以为流水线就成立了。

不一定。

如果前一个 Agent 输出：

```text
问题大概在 auth 模块。
```

Analyzer 还是需要重新：

- 搜索 auth 文件；
- 找到函数；
- 确认调用链；
- 排除无关文件。

这相当于 Locator 没完成工作。

好的交接应该是：

```text
Primary suspect:
src/auth/token.js:validateToken:42-68

Symptoms:
刷新 token 后，第一次 API 请求偶尔返回 401。

Evidence:
1. refreshToken() 会异步更新 tokenStore。
2. request() 可能在 tokenStore 更新前读取旧 token。
3. 失败日志中的 jti 与最新 token 不一致。

Already excluded:
- JWT 过期时间配置正常。
- 服务端时钟偏差小于 1 秒。
- 失败请求不是 refresh token 过期。

Related files:
- src/auth/token.js
- src/api/request.js
- src/store/session.js
```

Analyzer 拿到后可以直接开始根因分析，而不是重新定位。

因此，判断交接契约是否合格，可以用一个简单标准：

> 下一个 Agent 是否能够在不重复上一阶段主要工作的前提下开始执行？

如果不能，交接就不合格。

## 十一、失败时不要让 Fixer 一直“再试一次”

假设 Verifier 运行测试失败。

最差的处理方式是：

```text
Verifier：测试失败
    ↓
Fixer：再改一次
    ↓
Verifier：还是失败
    ↓
Fixer：继续猜
```
这很容易形成死循环。

正确流程应该是：

```text
Verifier 产生新证据
    ↓
回退到 Analyzer
    ↓
重新分析根因
    ↓
生成新修复方向
    ↓
Fixer 再执行
```

原因很直接：

- Fixer 负责执行方案；
- Analyzer 负责判断方案；
- 测试失败说明判断可能有问题；
- 因此新证据应该交给 Analyzer，而不是让 Fixer自由猜测。

推荐增加重试规则：

```text
同一个阶段最多重试 2 次。

Verifier 首次失败：
回退 Analyzer，结合失败日志重新分析。

第二次仍失败：
停止自动修改，输出人工调查报告。

禁止：
Fixer 和 Verifier 无限循环。
```

这才是可控的自动化，而不是让 AI 不停试错。

## 十二、并行和流水线不是二选一

真实项目中，更常见的是混合模式。

### 模式一：先并行探索，再进入修复流水线

这是最适合陌生项目的方式。

```text
        ┌─ auth-explorer ─┐
任务 →  ├─ db-explorer ───┤
        └─ api-explorer ──┘
                 ↓
          主对话综合架构
                 ↓
            发现具体问题
                 ↓
Locator → Analyzer → Fixer → Verifier
```

可以输入：

```text
先并行调用 auth-explorer、db-explorer 和 api-explorer，
输出整体架构与跨模块调用链。

综合完成后，重点检查用户 role 的传递链路。

如果发现明确 Bug：
1. 进入 bug-locator
2. 进入 bug-analyzer
3. Analyzer 后暂停审批
4. 确认后执行 fixer
5. 最后 verifier 验证
```

这套模式非常适合：

- 新接手项目；
- 遗留系统排查；
- 跨模块线上问题；
- 大型重构前分析。

### 模式二：流水线中插入并行分析

某个 Bug 已经定位到一个函数，但你希望从多个角度检查：

```text
Locator
   ↓
   ├─ security-analyzer
   ├─ performance-analyzer
   └─ compatibility-analyzer
            ↓
       主对话综合
            ↓
          Fixer
            ↓
         Verifier
```

例如：

```text
bug-locator 已经定位到 validateToken 函数。

请并行从三个维度分析：
1. 安全风险
2. 并发和性能问题
3. 对现有调用方的兼容性影响

三份报告完成后，由主对话综合出一个修复方案。
先向我展示综合结论，确认后再调用 bug-fixer。
```

这就是：

> Pipeline + Parallel Stage。

### 模式三：多个独立 Bug 分别走流水线

例如：

```text
Bug A：购物车空数组报错
Bug B：订单金额字符串相加
```

两者互不相关，可以分别执行：

```text
流水线 A：
Locator A → Analyzer A → Fixer A → Verifier A

流水线 B：
Locator B → Analyzer B → Fixer B → Verifier B
```

但这里必须额外小心：

- 两条流水线是否修改同一文件；
- 是否使用同一工作区；
- 是否可能产生 Git 冲突；
- 测试是否相互影响；
- 是否应该分别创建分支或 Worktree。

因此，在真实项目中，不建议让两个 Fixer 同时修改同一个工作区。

更稳妥的方式是：

```text
调查与分析可以并行
代码修改要隔离工作区
最终统一做集成测试
```

## 十三、什么时候不应该使用 Sub Agent

不要把 Sub Agent 当成所有任务的默认答案。

下面这些场景，直接让主对话处理通常更简单。

### 1. 已知位置的简单错误

例如：

```text
src/utils/date.ts 第 32 行类型错误，帮我修复。
```

没有必要再跑四阶段流水线。

### 2. 单文件的小范围理解

例如：

```text
解释一下这个 useRequest hook。
```

单个主对话已经足够。

### 3. 输出无法独立交付
如果你拆出的子任务必须频繁互相询问，说明边界不合理。

### 4. 拆分成本高于任务成本
一个 5 分钟能解决的问题，不值得配置 4 个 Agent、3 份交接契约和一套审批流程。

Sub Agent 适合解决的是：

> 高复杂度、高噪声、强流程、可分工的任务。

不是为了展示架构而架构。

参考并行探索项目也明确指出，简单单文件查询和普通 Grep 搜索并不适合使用并行探索。

Bug 修复项目同样区分了适用场景：复杂 Bug 适合流水线，位置明确的简单 Bug 直接修即可。

## 十四、从 Demo 走向企业使用，还缺什么

参考项目适合帮助我们理解机制，但不能直接等同于企业级自动修复系统。

真实项目至少还需要补上以下能力。

### 1. 持久化阶段产物

不要只依赖聊天记录。

建议把每个阶段报告保存下来：

```text
.ai-tasks/
└── BUG-30494/
    ├── 01-locator.md
    ├── 02-analyzer.md
    ├── 03-fixer.md
    ├── 04-verifier.md
    └── metadata.json
```

这样可以：

- 审计每个阶段；
- 中断后恢复；
- 追踪错误决策；
- 复用前序结论；
- 生成 MR 描述。

### 2. 使用独立 Git 分支

Fixer 修改前创建分支：

```bash
git checkout -b fix/bug-30494
```

不要直接在主分支修改。

### 3. 增加修改范围限制

例如：

```text
只允许修改：
src/auth/token.ts
tests/auth/token.test.ts

不得修改：
package.json
构建配置
数据库迁移
公共 API 签名
```

### 4. 真实验证，而不是语言判断

Verifier 至少应该执行项目已有的：

```bash
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

具体执行哪些命令，应由项目情况决定。

只读代码后说“看起来没问题”，不能算验证通过。

### 5. 增加人工审批门

建议至少保留两个审批点：

```text
Analyzer → Fixer
Verifier → Merge
```

即：

- 根因确认后才能改；
- 验证通过后也不能自动合并，仍需代码审查。

### 6. 加入失败上限

例如：

```text
单阶段最多重试 2 次
整条流水线最多回退 1 次
超过上限转人工处理
```

### 7. 防止多个任务相互污染

同时执行多个 Bug 修复时，应使用：

- 独立分支；
- 独立 Git Worktree；
- 独立日志目录；
- 独立任务状态；
- 必要时增加并发锁。

否则两个 Fixer 可能同时修改同一工作区。

## 十五、一个更适合真实项目的最终编排模板

下面这段 Prompt 可以直接作为实践起点。

```markdown

- 编号：BUG-30494
- 症状：用户登录后偶尔 token 验证失败

- 复现步骤：
- 已知线索：

## 编排方式

### 阶段一：定位

调用 bug-locator。

要求：

- 只读
- 不修改代码
- 输出具体文件、函数、行号、证据、调用链
- 输出完整 Locator → Analyzer Handoff

### 阶段二：分析

调用 bug-analyzer，并将 Locator 报告作为输入。

要求：

- 找到根本原因
- 区分症状和根因
- 评估影响范围
- 提供最多三个修复方案
- 推荐一个方案
- 输出完整 Analyzer → Fixer Handoff
- 不修改代码

### 审批门

Analyzer 完成后停止。

向我展示：

1. 根本原因
2. 支持证据
3. 推荐方案
4. 预计修改文件
5. 风险
6. 验收标准

我明确确认后才能进入下一阶段。

### 阶段三：修复

调用 bug-fixer。

要求：

- 只实施已确认方案
- 最小必要修改
- 不重构无关代码
- 不增加无关依赖
- 输出 Diff、风险、回滚方案
- 输出完整 Fixer → Verifier Handoff

### 阶段四：验证

调用 bug-verifier。

要求：

- 检查 Diff 是否对应根因
- 运行相关测试
- 运行项目完整检查
- 给出执行命令和退出码
- 明确输出 PASS 或 FAIL
- 不得用“看起来正确”代替测试证据

## 失败处理

- Locator 证据不足：重新定位一次。
- Analyzer 根因不明确：停止并请求更多信息。
- Verifier 失败：把失败证据退回 Analyzer 重新分析。
- 禁止 Fixer 在没有新根因分析的情况下反复修改。
- 任一阶段重试超过 2 次，停止自动处理并输出人工调查报告。
```

## 十六、最终理解：Sub Agent 的本质是工程分工

学完并行探索和流水线编排后，不应该只记住：

```text
并行启动三个 Agent。
```

或

```text
创建 Locator、Analyzer、Fixer、Verifier。
```

真正应该建立的是下面这套认知。

### 并行探索解决的是吞吐量问题

```text
多个独立任务
    ↓
同时调查
    ↓
主对话综合
```

它的关键不是 Agent 数量，而是：

- 任务是否独立；
- 职责是否清楚；
- 输出粒度是否一致；
- 最终是否有综合阶段。

### 流水线解决的是过程控制问题

```text
定位
 ↓
分析
 ↓
确认
 ↓
修复
 ↓
验证
```
它的关键不是流程看起来漂亮，而是：

- 阶段之间是否真的有依赖；
- 每一步是否有清晰职责；
- 权限是否逐步放开；
- 交接信息是否完整；
- 失败是否能够回退；
- 关键节点是否允许人工介入。

### 主对话不是普通执行者，而是编排者

主对话负责：

```text
拆任务
分配 Agent
收集报告
连接信息
审查结论
控制权限
决定继续、回退或停止
```

Sub Agent 负责局部专业执行。

因此可以用一句话概括：

> Sub Agent 不是把一个任务交给更多 AI，而是把一项复杂工作改造成职责清晰、权限隔离、结果可审查的工程流程。

这才是并行探索和流水线编排真正值得学习的地方。
